# Components

## Button

何时使用：

- 触发明确命令。
- 保存、刷新、导出、确认。

类型：

- Primary：页面主操作，一个页面最多 1 个主要 Primary。
- Default：普通操作。
- Danger：删除、真实下单、高风险动作。
- Text：低优先级操作。

避免：

- 多个 Primary 并列。
- 危险操作和普通保存同视觉层级。

推荐组合：

- 主按钮 + 次按钮。
- 危险按钮 + Modal 确认。

## Input

何时使用：

- 文本、数字、配置字段。

规则：

- 数字字段优先使用数字输入。
- 密钥字段使用 password 类型并脱敏。
- Placeholder 只做示例，不替代 Label。

## Select

何时使用：

- 选项有限且互斥。

避免：

- 只有 2 个布尔值时使用 Select，优先用 Switch/Checkbox。
- 选项超过 20 个且无搜索。

## DatePicker

何时使用：

- 日志时间范围。
- 报表时间范围。

规则：

- 默认提供快捷范围：今天、近 7 天、近 30 天。
- 不让用户输入模糊日期。

## Tag

何时使用：

- 状态、类型、交易所、角色。

规则：

- 状态 Tag 必须颜色稳定。
- 不用 Tag 展示长文本。

## Badge

何时使用：

- 待处理数量。
- 异常数量。

避免：

- 普通数量展示。
- KPI 指标。

## Card

何时使用：

- KPI。
- 页面分区。
- 复杂配置组。
- 关系或结构视图：少量对象的主从关系、链路、配对关系、流程状态。

避免：

- 卡片套卡片。
- 把每一行数据做成卡片。
- 用卡片承载大量需要排序、筛选、批量操作的数据。

推荐组合：

- Card + KPI。
- Card + Table。
- Card + Form section。
- Card + Relationship view。

## Tabs

何时使用：

- 同一对象的多视图。
- 同一页面内平级内容切换。

避免：

- 用 Tabs 替代主导航。
- Tabs 超过 5 个。

## Table

何时使用：

- 账号列表。
- 策略单元管理列表。
- 交易对。
- 告警日志。
- 保险节点。
- 需要排序、筛选、比较、批量操作的数据。

不强制使用：

- 当页面目标是帮助用户理解少量对象之间的关系、层级、流向或配对状态时，可使用卡片、关系图或分组详情视图。
- 同一对象可按任务切换展示方式：管理和批量处理用 Table，总览和关系理解用 Card / Relationship view。

推荐组合：

- Table + Toolbar。
- Table + Filter。
- Table + Row Actions。

## Form

何时使用：

- 新增账号。
- 编辑策略。
- 运行配置。
- 高级参数。

推荐组合：

- Form + Alert。
- Form + Divider。
- Form + Submit Bar。

## 设计原则总结

- 任务优先：页面先服务用户判断和操作，不做装饰优先。
- 风险优先：真实下单、阻断、失败、删除必须醒目。
- 信息去重：同一状态只出现在最合适的位置。
- Token 优先：颜色、字体、间距、圆角、阴影都使用统一 Token。
- 组件优先：优先使用 Design System 中定义的组件组合，不临时发明 UI。
- 可访问优先：状态不能只靠颜色表达，Focus 和错误提示必须清楚。

## 页面设计检查清单

- 页面是否符合 Sidebar / Header / Content / Feedback 分层。
- Header 是否没有重复 KPI 卡片信息。
- 首屏是否能判断运行模式、配置状态和运行健康。
- 颜色是否全部来自 `color.md`。
- 字体层级是否符合 `typography.md`。
- 间距是否来自 `spacing.md`。
- 表格、表单、反馈是否分别符合对应规范。
- 空状态是否说明下一步。
- 真实下单和危险操作是否醒目。

## 前端开发检查清单

- 新页面开发前是否阅读对应规范文档。
- 是否复用现有 helper、CSS class 和组件模式。
- 是否没有新增未定义颜色、字号、间距。
- 是否没有破坏侧边栏收起/展开。
- 是否没有把敏感配置提交到 Git。
- 是否没有改变业务逻辑，除非需求明确。
- 是否通过编译和单元测试。
- README 是否在 UI/功能/启动方式变化后同步更新。

## AI 页面生成检查清单

- 生成前是否读取 `docs/design-system/*`。
- 是否明确页面类型：Dashboard、Table、Form、Detail、Config。
- 是否使用统一浅色 SaaS 后台风格。
- 是否避免重复展示同一状态。
- 是否使用规定 Token，不自创颜色和间距。
- 是否根据任务选择展示方式：批量管理用表格，关系理解用卡片或关系视图。
- 是否为危险操作设计二次确认。
- 是否为 Empty、Loading、Error 设计状态。
- 是否输出后自检与 Design System 的偏差。
