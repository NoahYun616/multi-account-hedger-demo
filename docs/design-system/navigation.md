# Navigation

## 信息架构原则

导航按用户任务组织，而不是按代码模块或配置文件组织。

一级导航控制在 6 到 9 项。

默认一级导航：

- 总览
- 对冲监控
- 账号
- 策略
- 交易对
- 配置体检
- 原始配置
- 使用说明

后续可新增：

- 保险节点
- 告警记录
- 策略结果

## 菜单层级规范

默认只使用一级菜单。

二级菜单仅当某一领域页面超过 3 个时使用，例如：

- 策略
  - 策略单元
  - 跟随关系
  - 风控规则

禁止超过 2 层菜单。

## Breadcrumb 规范

仅在以下情况使用面包屑：

- 页面层级超过 2 层。
- 用户从列表进入详情页。
- 页面不是左侧导航直接可达。

示例：

- 策略 / unit_1 / BTC-USDT
- 账号 / websea_master_1

不需要：

- 总览。
- 一级导航页面。

## 导航命名规范

命名必须短、稳定、任务导向：

- “账号”优于“账号管理”。
- “策略”优于“策略配置”。
- “交易对”优于“交易对配置”。
- “配置体检”优于“一键检查”。

避免：

- 技术模块名。
- 文件名。
- 动词堆叠。

## 当前状态与导航关系

Sidebar 不展示运行状态。状态属于页面内容或 Header。

允许在菜单项上使用 Badge：

- 告警记录：未处理数量。
- 配置体检：阻断数量。
- 保险节点：异常数量。

Badge 不用于普通数量展示。

## Navigation 检查项

- 导航是否按用户任务组织。
- 是否没有超过 2 层。
- 命名是否短且稳定。
- Sidebar 是否没有重复主内容状态。
- Badge 是否只用于待处理或异常数量。
