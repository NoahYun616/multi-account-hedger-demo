# 合约保险对冲工具优化迭代方案

## 一、文档目的

合约保险对冲工具已经完成基础部署，当前主要能力是通过 Gate 与 Websea 两个平台之间的仓位同步，辅助用户执行合约保险相关对冲策略。

本文件用于明确下一阶段产品优化方向、功能新增范围、风控要求，以及需要 Websea 技术团队支持的接口和系统能力。后续我们会基于这份文档继续拆解产品需求、研发排期和测试验收标准。

## 二、当前工具定位

该工具是 Websea 合约保险业务的配套对冲工具，当前核心逻辑为：

- 用户在 Gate 主账号手动开仓、平仓或调仓。
- 工具监听 Gate 主账号仓位变化。
- 工具根据策略配置，在 Websea 自动建立同向或反向仓位。
- 工具按仓位同步，不按订单逐笔复制。
- 工具支持 Gate 子账号跟随 Gate 主账号，也支持 Websea 子账号在 Websea 主账号成功后跟随。

当前工具更准确的定位是：

> Websea 合约保险用户的跨交易所仓位同步与风险控制工具。

它不是行情判断工具，也不是收益承诺工具。它的核心价值是把用户设定好的对冲规则稳定执行出来，并逐步补齐资金、杠杆、强平、保险节点和异常状态的风险管理能力。

## 三、当前主要问题

### 1. Websea 资金不足时缺少前置保护

如果 Websea 侧资金不足，可能出现：

- Gate 已经开仓，但 Websea 未能成功建立对冲仓位。
- Websea 只开出部分仓位。
- 工具持续重试或只在日志中记录失败。
- 用户误以为对冲仍然完整。

这会导致策略从“对冲状态”变成“单边暴露状态”，对合约保险策略的执行和风险控制都会造成影响。

### 2. 两边杠杆不一致时存在强平断腿风险

Gate 与 Websea 两边杠杆可以不一致。对冲效果的核心不是杠杆一致，而是实际仓位价值尽量匹配。

但如果某一边杠杆过高、保证金缓冲不足，就可能先被强平。一旦一边仓位消失，另一边仓位还在，用户会立即暴露在单边行情风险中。

因此，工具需要把“强平前预警”和“强平后自动处理”作为核心风控能力。

### 3. 当前工具没有把当前账号保险节点状态纳入决策

当前工具主要处理交易仓位，没有系统性纳入合约保险中的关键变量，例如：

- 保费扣除。
- 节点生成。
- 节点冻结。
- 节点失效。
- 空投到账周期。
- 已领取和未领取空投。
- 风控节点。
- 当前账号保险开关和保险池类型。

这会导致用户只能看到仓位是否同步，却无法判断当前 Websea 账号的保险链路是否正常，例如保险是否开启、节点是否有效、是否存在冻结/失效/风控节点，以及是否适合继续用这个账号执行对冲。

### 4. 当前异常状态对用户提醒不够强

对冲工具运行中最重要的风险，不是单次收益低，而是用户不知道风险已经发生。

典型场景包括：

- Websea 开仓失败。
- Websea 资金不足。
- 两边仓位偏差过大。
- 一边仓位异常消失。
- 一边 API 无法读取仓位。
- 工具处于真实下单模式但用户没有明显感知。

这些状态必须从“日志可见”升级为“前端显著可见 + 告警触达 + 策略自动保护”。

## 四、优化目标

### 目标一：保障对冲完整性

工具必须持续回答：

- Gate 当前仓位是多少。
- Websea 目标仓位是多少。
- Websea 实际仓位是多少。
- 当前对冲完成率是多少。
- 当前未对冲敞口是多少。
- 偏差是否超过风控阈值。

### 目标二：避免对冲断腿

我们要重点防止：

- Gate 已开仓，Websea 没跟上。
- Websea 已强平，Gate 还在。
- Websea 下单失败，Gate 仍然单边暴露。
- 两边仓位偏差持续扩大。
- API 读取失败后工具继续盲目运行。

### 目标三：把当前账号保险节点状态纳入策略判断

合约保险对冲工具不能只看交易仓位，还要把当前 Websea 账号的保险节点状态纳入策略判断。

工具后续需要同时呈现：

- 当前账号是否开启保险。
- 当前账号使用的保险池类型。
- 保险账户余额。
- 保费扣除记录。
- 节点生成情况。
- 节点有效性。
- 冻结、失效、风控节点情况。
- 已领取和未领取空投。
- 最近一次节点生成、冻结、失效或风控原因。

工具不负责计算资金费率、持仓时间成本或未来综合收益，也不对用户是否继续使用该账号对冲做收益承诺。用户根据自己的资金成本、持仓周期和节点状态自行判断。

### 目标四：形成可规模化运营的工具能力

如果该工具后续要服务更多 Websea 合约保险用户，我们需要让它具备稳定的配置检查、风险预警、异常处理和策略复盘能力，而不是只面向少数熟练用户使用。

## 五、功能优化与新增需求

## P0：优先建设的安全能力

### 1. 对冲完整性检测

前端需要实时展示：

- Source 当前仓位。
- Hedge 目标仓位。
- Hedge 实际仓位。
- 对冲完成率。
- 仓位偏差数量。
- 仓位偏差比例。
- 未对冲敞口。

示例：

> Gate BTC 多仓名义价值 10,000 USDT，Websea 目标空仓名义价值 10,000 USDT，Websea 实际空仓名义价值 7,000 USDT，当前对冲完成率 70%，未对冲敞口 30%。

风控规则建议：

- 偏差小于阈值：正常。
- 偏差超过阈值但持续时间较短：黄色预警。
- 偏差超过阈值并持续 N 秒：红色告警。
- 偏差超过极限阈值：暂停策略并触发自动保护。

### 2. Websea 资金不足预检查

在 Websea 自动开仓前，工具需要预先计算：

- Websea 合约账户可用余额。
- 目标仓位名义价值。
- 当前设置杠杆下预计占用保证金。
- 开仓后剩余保证金缓冲。
- 是否满足最低安全缓冲。

处理逻辑：

- 资金充足：正常开仓。
- 资金略不足：提示调整杠杆、降低仓位或补充资金。
- 资金严重不足：阻止开仓并报警。
- Gate 已经有仓位但 Websea 无法安全对冲：提示用户减少 Gate 仓位或补充 Websea 保证金。

这里不建议为了开仓而无脑提高杠杆。杠杆调整必须受安全上限控制。

### 3. 对冲断线保护

新增“对冲断线保护”能力。

触发条件包括：

- 一边仓位异常变为 0，另一边仍有仓位。
- 一边疑似强平。
- 一边开仓失败，另一边已成交。
- 两边仓位偏差超过阈值并持续 N 秒。
- Websea API 或 Gate API 长时间无法读取仓位。

触发后动作：

1. 暂停该策略单元继续开新仓。
2. 前端红色告警。
3. 发送 Telegram 或其他通知。
4. 自动使用 reduce-only 平掉另一边仓位。
5. 平仓失败时持续重试。
6. 将策略单元标记为异常状态，等待人工确认后恢复。

### 4. 强平风险监控

工具需要监控两边仓位的强平风险。

建议展示：

- 当前标记价格。
- 开仓均价。
- 强平价格。
- 距离强平百分比。
- 保证金率或风险率。
- 当前杠杆。
- 当前保证金模式。

风险分层：

- 安全：距离强平较远。
- 预警：距离强平较近，提示补充保证金或降低仓位。
- 危险：距离强平极近，建议自动减仓或暂停策略。
- 已强平：立即执行对冲断线保护。

### 5. 真实下单模式强提醒

当前 `dry_run` 是核心安全开关。

前端需要永久展示当前运行模式：

- 模拟运行：绿色或灰色提示。
- 真实下单：红色提示。

从模拟切换到真实下单前，需要二次确认，并自动执行配置检查：

- API 是否填写完整。
- API 是否无提币权限。
- Websea 资金是否充足。
- 交易对配置是否完整。
- ratio 和合约面值是否配置。
- 最大仓位和最大调整量是否配置。
- 对冲断线保护是否开启。

## P1：提升易用性与稳定性的功能

### 6. 安全模式

新增“安全模式”。

安全模式开启后：

- 默认开启对冲完整性检测。
- 默认开启对冲断线保护。
- 默认开启强平风险预警。
- 默认限制最大杠杆。
- 默认限制单币种最大仓位。
- 默认限制最大未对冲敞口。
- Websea 资金不足时不强行开仓。
- API 读取异常时暂停策略。

建议安全模式默认配置：

```text
最大杠杆：可配置，例如 20x / 50x
最大单币种仓位：可配置
最大未对冲敞口：例如 5%
最大未对冲持续时间：例如 3 秒
资金不足处理：暂停并告警
强平风险过高处理：减仓或暂停
```

### 7. Websea 杠杆推荐与自动调整

工具可以根据 Websea 可用资金和目标仓位，计算推荐杠杆。

推荐逻辑：

1. 计算目标仓位名义价值。
2. 读取 Websea 可用保证金。
3. 根据当前杠杆计算预计占用保证金。
4. 判断开仓后是否保留足够保证金缓冲。
5. 如果当前杠杆不够，计算最低可用杠杆。
6. 如果最低可用杠杆低于安全上限，可以提示或自动调整。
7. 如果超过安全上限，则不自动开仓，并提示资金不足。

核心原则：

> 自动调整杠杆是风控辅助，不是强行开仓手段。

### 8. 仓位价值换算展示

前端需要同时展示张数和实际仓位价值。

建议展示：

- Gate 张数。
- Gate 合约面值。
- Gate 实际币数量。
- Gate 仓位名义价值。
- Websea 张数。
- Websea 合约面值。
- Websea 实际币数量。
- Websea 仓位名义价值。
- 当前换算比例。

这样可以避免用户只看张数，而忽略两边合约面值不同导致的对冲偏差。

### 9. 保证金模式解释与推荐

前端在选择 `cross` 和 `isolated` 时，需要直接显示解释：

- `cross`：全仓模式，账户可用余额共同承担风险，仓位不容易单独强平，但可能影响账户更多资金。
- `isolated`：逐仓模式，单仓风险隔离，但更容易强平。

在对冲工具场景下，我们默认推荐：

- 对冲稳定优先：使用 cross。
- 风险隔离优先：可选择 isolated，但必须提示更容易强平。

### 10. 配置体检

新增“一键配置体检”。

检查项包括：

- 账号 API 是否可用。
- Websea API 是否有交易权限。
- API 是否未开启提币权限。
- Gate WebSocket 是否可连接。
- Websea 仓位接口是否可读取。
- 交易对是否存在。
- ratio 是否配置。
- 合约面值是否配置。
- 杠杆是否在交易所允许范围内。
- Websea 可用资金是否足够。
- 当前是否开启对冲断线保护。
- 当前是否开启告警。

体检结果分为：

- 通过。
- 警告。
- 阻断。

阻断项未解决前，不允许切换到真实下单模式。

## P2：当前账号保险节点状态增强

### 11. 当前 Websea 账号保险节点状态接入

工具需要接入当前 Websea 账号的合约保险节点数据。

建议展示：

- 当前账号使用的保险池类型。
- 是否开启保险。
- 保险账户余额。
- 已生成节点。
- 有效节点。
- 冻结节点。
- 失效节点。
- 已领取空投。
- 未领取空投。
- 风控节点。
- 预计空投到账时间。
- 最近一次节点生成、冻结、失效或风控原因。

这样用户可以看到当前 Websea 账号是否真的进入保险节点链路，而不是只看到交易侧仓位。

本工具不展示平台保险池健康度，不替平台判断保险池承压情况，也不限制用户真实生成保险节点和支付保费。工具只呈现当前账号相关的保险节点状态与可用性，供用户自行判断是否继续用该账号执行对冲。

### 12. 策略已知结果汇总

每轮策略结束后，工具自动生成已知结果汇总。

汇总内容：

- Gate 开仓和平仓记录。
- Websea 开仓和平仓记录。
- Gate 盈亏。
- Websea 盈亏。
- Websea 保费。
- 保险节点数量。
- 预计空投。
- 实际空投。
- 当前节点状态。
- 是否出现对冲偏差。
- 是否出现接口异常。
- 是否触发风控保护。

结果汇总用于用户查看本轮已知执行结果，也可作为后续优化工具体验和风控策略的数据基础。

该汇总不计算资金费率，不计算持仓时间成本，不预测未来节点收益，不输出最终净收益结论。用户自行结合资金成本、持仓周期、保险节点状态和自身风险偏好判断策略是否继续。

## 六、需要 Websea 技术团队支持的能力

Gate 侧只能基于现有 API 和交易所已有能力使用，本阶段不要求 Gate 提供任何定制支持。

本工具后续增强主要依赖 Websea 技术团队提供更完整、更稳定的接口能力。

### 1. Websea 仓位接口增强

需要接口稳定返回：

- 多仓数量。
- 空仓数量。
- 净仓位。
- 开仓均价。
- 标记价格。
- 强平价格。
- 保证金模式。
- 杠杆。
- 已用保证金。
- 未实现盈亏。
- 保证金率或风险率。

尤其需要清晰区分：

```text
long position
short position
net position
```

### 2. Websea 可用余额与保证金接口

需要接口返回：

- 合约账户总权益。
- 可用保证金。
- 已占用保证金。
- 冻结保证金。
- 保险账户余额。
- 是否可开目标仓位。
- 预计开仓所需保证金。

这部分用于资金不足预检查和开仓前风控。

### 3. Websea 杠杆查询与设置接口

如果要做杠杆推荐和自动调整，需要 Websea 支持：

- 查询当前杠杆。
- 设置某交易对杠杆。
- 查询某交易对最大可用杠杆。
- 查询当前用户是否允许设置该杠杆。
- 返回设置失败的标准化错误码。

如果 Websea 暂时不支持自动设置杠杆，工具先做“杠杆推荐 + 人工确认”。

### 4. Websea reduce-only 平仓能力

对冲断线保护强依赖 reduce-only。

需要 Websea 下单接口支持：

- 只减仓，不开反向仓。
- 市价 reduce-only 平仓。
- 指定多仓或空仓方向平仓。
- 返回明确成交状态。

这是避免“本来要平仓，结果反手开仓”的关键能力。

### 5. Websea 强平与风险状态接口

需要支持：

- 查询强平价。
- 查询距离强平百分比。
- 查询仓位风险等级。
- 查询仓位是否已强平。
- 查询是否发生系统减仓或自动平仓。

最好支持 WebSocket 推送。如果暂时不支持推送，至少需要稳定轮询接口。

### 6. Websea 订单状态与成交回报

需要稳定返回：

- 订单是否提交成功。
- 是否成交。
- 成交数量。
- 平均成交价。
- 剩余未成交数量。
- 失败原因。
- 是否部分成交。

如果只有“下单成功”而没有成交明细，工具无法准确判断对冲是否真正完成。

### 7. Websea 合约保险数据接口

为了让工具成为合约保险策略工具，需要 Websea 提供：

- 当前用户是否开启保险。
- 当前保险池类型。
- 保险账户余额。
- 保费扣除记录。
- 节点生成记录。
- 节点状态。
- 节点冻结原因。
- 节点失效原因。
- 空投预计金额。
- 空投领取状态。
- 是否命中风控。

这部分是工具从“仓位同步工具”升级为“合约保险账号状态控制台”的关键。

接口只需要服务当前账号维度的保险状态展示，不要求提供平台保险池健康度、保险池承压情况或平台级风控边界。

### 8. Websea 标准化错误码和告警事件

建议 Websea 技术侧统一返回标准化错误码：

- 余额不足。
- 杠杆不合法。
- 超过最大持仓。
- 超过最大下单量。
- 仓位不存在。
- 仓位已强平。
- reduce-only 无可减仓位。
- API 权限不足。
- 交易对不可用。
- 保险账户状态异常。

工具需要根据错误码执行不同动作，而不是通过错误文案猜测原因。

## 七、Gate 侧能力边界

Gate 侧只基于已有功能使用，不要求 Gate 提供技术支持。

当前可依赖：

- Gate API。
- Gate Futures WebSocket。
- Gate 仓位查询。
- Gate 下单和平仓。
- Gate 合约账户余额。
- Gate 已有保证金和杠杆设置能力。

需要注意：

- Gate 主账号通常由用户手动操作，工具只是监听。
- Gate 主账号实际杠杆以用户在 Gate 页面设置为准。
- Gate 子账号如果由工具跟随，才受工具配置里的 source 杠杆影响。
- 所有产品化适配和策略闭环，应优先在工具和 Websea 侧完成。

## 八、建议迭代节奏

### 第一阶段：安全可用

- 真实下单模式强提醒。
- 一键配置体检。
- Websea 资金不足预检查。
- 对冲完整性检测。
- 仓位价值换算展示。
- 对冲断线报警。

### 第二阶段：自动风控

- 强平风险监控。
- 一边强平后自动平另一边。
- reduce-only 紧急平仓。
- 策略单元自动暂停。
- Telegram 或其他告警触达。

### 第三阶段：安全模式

- 最大杠杆限制。
- 最大仓位限制。
- 最大未对冲敞口限制。
- Websea 杠杆推荐。
- 资金不足时给出操作建议。
- cross / isolated 解释和推荐。

### 第四阶段：保险策略增强

- 保险节点数据接入。
- 当前账号保险状态展示。
- 策略已知结果汇总。
- 用户账号节点状态分析。

## 九、最终产品方向

合约保险对冲工具不应停留在“能同步仓位”的技术工具阶段。

我们要把它升级为：

> Websea 合约保险账号状态与对冲风险控制台。

它需要同时回答三个问题：

1. 仓位有没有同步成功？
2. 风险有没有失控？
3. 当前 Websea 账号的保险节点状态和本轮已知执行结果是什么？

只有同时解决这三个问题，工具才具备规模化给 Websea 合约保险用户使用的基础。
